# 仕様改善提案 — スライドからの見直し

現スライド（PwC「統合自律型のデータ分析監査」）は方向性・分析手法は完成度が高い。一方、**実装・運用・商用化の段階で詰まる論点**が未記述。ここでは「課題（現状の不足）→ 改善仕様 → 効果」で整理し、各改善を `docs/` `config/` のどこに反映したかを示す。

優先度: ◎=実装/運用の前提（必須） / ○=品質・信頼性に直結 / △=拡張・差別化

---

## ◎ 1. コスト・スケールのファネルを明示
- **課題**: 「全件網羅」と自律エージェント（LLM）を同列に語ると、数百万明細にLLMを流す非現実的な実装に見える。スライドにコスト構造の説明がない。
- **改善仕様**: ルール+MLを決定論的に全件適用 → 高リスク部分集合（数%）にのみエージェント深掘り、というファネルを設計の前提として明文化。閾値で網羅性とコストをトレードオフ可能に。
- **効果**: 「全件×現実的コスト」の両立根拠が示せる。提案時の最頻質問（コスト・スケーラビリティ）に即答できる。
- **反映先**: `docs/architecture.md` §3-4, `CLAUDE.md` §4.1

## ◎ 2. プロンプトインジェクションを脅威モデルに追加
- **課題**: エージェントはメール・領収書OCRという**当事者が操作可能な入力**を読むのに、検知回避攻撃（証憑への指示埋め込み）の記述がゼロ。不正検知システムとして致命的な盲点。
- **改善仕様**: 指示とデータの構造的分離、証憑からのツール発火禁止、出力スキーマ強制、注入パターン検知（検知自体を隠蔽の疑いとして扱う）、能力の技術的最小化。
- **効果**: 検知回避耐性。係争時に「AIが操作され見逃した」リスクを低減。
- **反映先**: `docs/security-privacy.md` §1, `docs/agent-design.md` §5

## ◎ 3. 従業員プライバシーの法的基盤を前提条件化
- **課題**: 「read-only・同意・監査ログ」だけでは、従業員のメール/予定表/移動記録の解析は運用できない。利用目的特定・労使協議・保存期間・本人通知・データ最小化が抜けると導入そのものが止まる。
- **改善仕様**: コネクタごとに利用目的・法的根拠・保存期間・通知状況を保持し未充足なら技術的にブロック。最小フィールド取得（本文は必要時のみG1承認下）。グローバル時はGDPR/DPIA対応。
- **効果**: 導入の法的・労務的ハードルを事前にクリア。最大の頓挫要因を潰す。
- **反映先**: `docs/security-privacy.md` §3, `CLAUDE.md` §2

## ◎ 4. 独立性の用途切替（Track A / B）
- **課題**: PwC ブランドで提供する以上、監査法人が被監査会社の統制を構築・運用すると独立性（公認会計士法第24条の2 等）を毀損しうる。用途の区別がスライドにない。
- **改善仕様**: `engagement_mode: track_a | track_b` を構成に持ち、track_b では是正提案を助言表現に制限・クライアント意思決定記録を必須化・独立性チェックリストを起動条件に。
- **効果**: 商用化（外部展開）と法定監査利用を、独立性を保ったまま両立。
- **反映先**: `docs/governance.md` §1

## ○ 5. 誤検知（アラート疲れ）管理
- **課題**: ルール+MLは大量のアラートを生む。トリアージ・較正・抑制の仕組みがないと「使われないダッシュボード」化する。スライドは検知力中心で運用負荷の記述が薄い。
- **改善仕様**: スコアによるトリアージ階層（自動棄却/人手レビュー/エスカレート）、各ルール/モデルの的中率計測、確定/棄却フィードバックによる既知良性パターンの抑制、誤検知注記（除外条件）をルールに必須化。
- **効果**: レビュー工数の集中配分。継続運用に耐える。
- **反映先**: `config/rules/rule_catalog.yaml`（各ルールの `false_positive_notes` / `risk_scoring`）, `docs/governance.md` §2

## ○ 6. モデルリスク管理（MLライフサイクル）
- **課題**: アルゴリズムとXAIは記載があるが、バリデーション・ドリフト監視・バイアス検査・再検証・バージョン管理がない。金融機関等では規制期待事項。
- **改善仕様**: 導入前検証、性能指標の継続計測、ドリフト監視と再較正トリガ、公平性検査、定期再検証、モデル/特徴量/閾値のバージョン紐付け。
- **効果**: 規制対応・監査対応の説明可能性。モデル劣化の早期検知。
- **反映先**: `docs/governance.md` §2

## ○ 7. 改ざん不能な監査ログ（証跡の強化）
- **課題**: 「証跡保全」はあるが、係争・不正調査に耐える完全性（改ざん検知）・来歴・再現性の要件が未定義。
- **改善仕様**: WORM＋ハッシュチェーン、全証憑への来歴付与、上書き禁止（追記のみ）、保存期間、後日の再現性。
- **効果**: 法的証拠能力・監査トレイルの担保。
- **反映先**: `docs/governance.md` §4, `config/schemas/data_contracts.json`（`AuditLogEntry`）

## ○ 8. データ品質・完全性照合（全件主張の裏付け）
- **課題**: 「全件」を謳う以上、対象データが本当に全件かを証明できねばならない。完全性照合・品質ゲートがスライドにない。
- **改善仕様**: 取込件数・金額合計をERP管理値と突合、欠損/重複/期ズレ検出、必須項目/型/参照整合性のゲート、OCR信頼度の保持とエスカレート。
- **効果**: 「全件網羅」の主張に裏付け。GIGO（ゴミ入力→ゴミ出力）回避。
- **反映先**: `docs/architecture.md` §4

## ○ 9. テスト戦略（合成不正による検出力評価）
- **課題**: 「検出できる」ことの検証方法がない。導入後に効くかを事前に測れない。
- **改善仕様**: 合成不正の注入（既知手口を人工生成しデータに混入）、過去確定事案のリプレイ、レッドチーム（インジェクション含む）による回避テスト。Precision/Recall・的中率で評価。
- **効果**: 導入前に検出力を定量化。継続的な精度改善の基盤。
- **反映先**: 本書 §下記「テスト戦略」, `docs/governance.md` §2

## △ 10. ナレッジ蓄積の正循環（学習ループ）
- **課題**: 「PwCのリスク検知ノウハウ蓄積」はあるが、確定事案がどうルール/シナリオに還流するかの仕組みが未記述。
- **改善仕様**: 確定不正 → 新ルール/シナリオ化 → ルールパック更新（バージョニング）、的中率に基づくルールの自動見直し、匿名化したクロスクライアント・パターン学習。
- **効果**: 使うほど賢くなる。差別化資産（ノウハウ）の体系的蓄積。
- **反映先**: `config/rules/rule_catalog.yaml`（メタデータ・バージョン）, `docs/architecture.md` §5-6

## △ 11. 他ドメインへの拡張性
- **課題**: 経費特化に見えるが、構造は仕訳・購買・給与に汎用化できる。スライドはその拡張余地を明示していない。
- **改善仕様**: 「ETL→ルール/ML/探索→エージェント検証→HITLレポート」をコアとし、費目別シナリオ＋コネクタの差し替えでドメイン展開。
- **効果**: 単一プロダクトから監査AIプラットフォームへ拡張。
- **反映先**: `docs/architecture.md` §6

## テスト戦略（§9 詳細）

| 種別 | 方法 | 指標 |
|---|---|---|
| 合成不正注入 | 16+ の手口（`fraud_scenarios.yaml`）をパラメタ化して人工生成、正常データに既知比率で混入 | Recall（注入を何%捕捉）、Precision |
| 過去事案リプレイ | 確定済み不正の過去データを再投入し再現を確認 | 再現率、見逃し分析 |
| 誤検知評価 | 正常データのみで実行し過検知率を測定 | False Positive Rate、上位N的中率 |
| 回避テスト（レッドチーム）| 証憑へのインジェクション・分割・カラ申請の偽装を試行 | 回避成功率（低いほど良い）|
| ドリフト | 期間をずらして再評価しスコア分布の安定性を確認 | 分布乖離度 |

合成不正生成器は `config/rules/fraud_scenarios.yaml` の各シナリオ定義（手口・検証データ）から導出する。これにより「検知ルール」と「テストケース」が単一定義から生成され、整合する。
