# ガバナンス仕様 — 独立性・モデルリスク・規制対応・監査ログ

スライドは検知能力（ルール/ML/エージェント）を厚く描く一方、**「この仕組みを誰が・どの立場で・どの規制下で使うか」**のガバナンスが手薄。実運用・商用化の可否を分けるのはこの層なので、ここを仕様化する。

## 1. 独立性 — 用途の区別（最重要・要法務確認）

本ソリューションは2つの用途で意味が大きく異なる。**構成で用途を明示的に切り替える**こと。

### Track A: 被監査会社の内部監査・不正調査支援（既定）
- クライアントの内部監査部門・不正調査チームが自社のために使用。
- AI は「内部統制の高度化ツール」。独立性の論点は相対的に小さい。
- 子会社・事業部管理によるモニタリング、アラート配信もこの文脈。

### Track B: 監査法人による外部商用化 / 法定監査での利用
- 監査法人が法定監査の手続として、あるいは非監査会社向けサービスとして提供する場合。
- **自己レビュー・自己利益の脅威**に注意。被監査会社の内部統制を監査法人が構築・運用すると独立性を毀損しうる（公認会計士法第24条の2 等の非監査証明業務の制限）。
- 監査ツールの「設計・提供」と「被監査会社における運用・是正の意思決定」は分離する。AI は評価・根拠提示まで、是正の意思決定はクライアント（または独立した立場）が行う構造が必須。

> 実装要件: `config` に `engagement_mode: track_a | track_b` を持ち、track_b では (a) 是正アクションの自動提案を「助言」表現に制限、(b) クライアント側の意思決定記録を必須化、(c) 独立性チェックリストの充足を起動条件とする。具体の可否・線引きは案件ごとに法務・品質管理部門の確認を前提とする。

### 配備レイヤ（3線 → 2線 → 1線）— 未決定・要方針

同じ分析エンジンを、統制の前段へ寄せていく発展構想がある。利用主体（Track A/B）とは別の第2の軸で、独立性にさらに影響する。

| 配備（線） | タイミング | 統制の性質 | AIの役割 | 強制力・HITL |
|---|---|---|---|---|
| 3線・内部監査 | 定期バッチ | 検知／独立保証 | 仮説生成〜深掘り判断 | フラグのみ。監査人が個別に最終判断 |
| 2線・継続モニタリング | 準リアルタイム | 検知／監視 | アラート生成・トリアージ | 例外ベース確認。フラグ＋通知 |
| 1線・現場 | リアルタイム（承認経路内） | 予防／実時間 | 即時スコア＋非同期で深掘り | 高精度な決定論ルールのみ硬ブロック。承認WF自体がHITL＋例外の非同期レビュー |

帰結:
- **独立性**: 1線・2線へ下りるほど「独立した保証」でなく経営者自身の統制になる。監査クライアントの現場統制を監査法人が作って回すと**自己レビューの脅威**が強まり、PIE（社会的影響度の高い事業体）相手では禁止域に近づく。緩和レバーは提供形態 ── 個社向けに設計・運用（bespoke）するほど自己レビュー寄り、製品として渡してクライアントが自ら設定・運用するほどベンダーに近く防御的。つまりプロダクト化は商機であると同時に独立性のミティゲーション設計でもある。
- **ファネルが成立条件に変わる**: 承認経路の中で全件に5フェーズ＋外部API照会を同期実行するのはレイテンシ・コスト両面で不可能。1線では決定論ルール＋学習済みMLスコアリング（ミリ秒）だけがインラインで回り、エージェントの深掘りは高リスク部分集合に対し非同期で走る分離が必須。
- **予防化が誤検知の経済性を変える**: 事後の検知フラグが外れてもレビュー工数増で済むが、実時間の予防ブロックが外れると業務を止める。よって1線の硬ブロックは「承認者欠落・ハード上限超過・制裁先」級の高精度な決定論ルールに限定し、確率的なもの（ML・エージェント）は最大でも警告/レビュー送りに留める階層的強制力が要る。同じルールでも姿勢ごとに強制アクションが変わるため、`rule_catalog.yaml` に「姿勢別の強制アクション」の軸を足すのが自然（スコープ確定後）。

> 実装方針（保留中）: `engagement_mode` を上表のデプロイ姿勢まで拡張する。ただし主たる射程 ── (a) まず内部監査（3線）で立ち上げ段階的に下ろす、か (b) 最初から1線リアルタイムを見据えプロダクト化・非監査エンティティ前提で土台を作る ── の決定が先。ここで独立性の書き方とアーキテクチャの初期重心が変わる。

## 2. モデルリスク管理（ML ライフサイクル・spec改善点）

スライドはアルゴリズム（Isolation Forest / LOF / Autoencoder / PCA）と XAI に触れるが、モデルの**ライフサイクル管理**がない。金融機関等ではモデルリスク管理が規制期待事項。

| 項目 | 要件 |
|---|---|
| バリデーション | 導入前に既知不正データ（合成＋過去事案）で検出力・誤検知率を評価。`docs/spec-improvements.md` のテスト戦略と連動 |
| 性能指標 | Precision / Recall / 適合率@上位N / アラート的中率 をルール・モデル単位で継続計測 |
| ドリフト監視 | 入力分布・スコア分布の変化を監視。閾値超で再学習・再較正をトリガ |
| バイアス検査 | 特定部門・職位・属性に検知が偏らないかを定期検査（公平性）|
| 説明可能性 | 全 ML 所見に SHAP 寄与上位を添付（根拠なきスコア禁止）|
| 再検証 | 定期（例:年次）＋重大変更時の再検証。検証結果を文書化・保管 |
| バージョン管理 | モデル・特徴量・閾値のバージョンを所見に紐付け（再現性）|

## 3. 規制・基準フレームワークへのマッピング

本ソリューションが支援する統制を、関連基準に対応づけて説明できるようにする（クライアント説明・監査対応で必須）。

| フレームワーク | 関連箇所 | 本ソリューションの寄与 |
|---|---|---|
| J-SOX（内部統制報告制度）| 業務プロセス統制・IT全般統制（ITGC）| 経費プロセスの統制逸脱・職務分掌違反（自己承認等）の全件検知 |
| 監査基準報告書240（不正に関連した監査人の責任）| 不正リスク要因の識別・対応 | 不正シナリオに基づく仮説生成と証憑検証 |
| 不正リスク対応基準 | 不正による重要な虚偽表示リスクへの対応 | 全件網羅による想定外リスクの発見 |
| COSO（内部統制・ERM）| 統制活動・モニタリング | 継続的モニタリングとアラート配信 |
| ACFE 不正の定義・分類 | 資産の不正流用（経費不正）| 費目別の手口分類（`fraud_scenarios.yaml`）|

注: 各基準の最新版・条番号は改正されうるため、運用前に最新の基準・ガイドラインを確認すること。

## 4. 改ざん不能な監査ログ（証跡保全の強化）

スライドの「証跡保全・説明可能性」を、係争・調査に耐えるレベルへ引き上げる。

### 要件
- **網羅性**: 入力明細 → 適用ルール/モデル（バージョン）→ 収集証憑（ID・取得時刻・コネクタ・取得者権限）→ スコア → 結論 → HITL判断 の連鎖を欠落なく記録。
- **完全性 (integrity)**: WORM ストレージ＋ハッシュチェーン（各レコードに前レコードのハッシュを含める）で改ざんを検知可能に。
- **来歴 (provenance)**: すべての証憑にソース・取得方法・取得時点を付与。二次加工はバージョンとして追記（上書き禁止）。
- **保存期間**: 規程・法令に基づき定義。経費・監査関連の保存義務年数を満たす。
- **再現性**: ログから、ある所見がなぜそのスコアになったかを後日再現できること。

スキーマは `config/schemas/data_contracts.json` の `AuditLogEntry`。

## 5. RACI（誰が何に責任を持つか）

| 活動 | AI エージェント | 監査人/調査担当 | 品質管理/法務 | クライアント経営 |
|---|---|---|---|---|
| 高リスク選別・仮説生成・証憑収集 | R | A | C | I |
| リスク評価の確定 | C | R/A | C | I |
| 是正・通報・処分の意思決定 | — | C | C | R/A |
| 独立性・プライバシー適合性判断 | — | C | R/A | I |
| モデル/ルールの承認・再検証 | — | C | R/A | I |

R=実行, A=説明責任, C=協議, I=情報共有。**AIが A（説明責任）を持つ行はない**。
