# アーキテクチャ仕様 — システム全体像

## 1. レイヤ構成

データ取込から統合レポートまでを、自律型 AI エージェントがオーケストレートする。下層ほど決定論的・低コスト、上層ほど推論的・高コスト。

```
┌─────────────────────────────────────────────────────────────┐
│  L6  HITL コンソール（監査人）                                  │
│      所見レビュー / 確定・棄却 / 是正起票 / フィードバック       │
├─────────────────────────────────────────────────────────────┤
│  L5  レポート生成                                              │
│      PDF（内部監査・証跡保全）/ PPTX（経営層報告）/ Excel・CSV  │
├─────────────────────────────────────────────────────────────┤
│  L4  自律エージェント・オーケストレータ                          │
│      観察→仮説生成→探索→検証→統合 のループ（agent-design.md）  │
│      ＋ 証憑コネクタ（read-only）                              │
├─────────────────────────────────────────────────────────────┤
│  L3  分析エンジン（並列）                                       │
│   ┌──────────────┬──────────────┬──────────────┐            │
│   │ ①ルールベース │ ②探索的分析   │ ③機械学習     │            │
│   │ 62ルール×10ｶﾃ│ 可視化/異常発見│ IF/LOF/AE/PCA │            │
│   │              │              │ ＋SHAP        │            │
│   └──────────────┴──────────────┴──────────────┘            │
├─────────────────────────────────────────────────────────────┤
│  L2  特徴量・正規化層                                          │
│      費目別特徴量空間 / 申請者行動プロファイル / マスタ結合      │
├─────────────────────────────────────────────────────────────┤
│  L1  データ取込・ETL（構造化＋非構造化）                        │
│      AIによる自動構造化 / データ品質・完全性照合                 │
└─────────────────────────────────────────────────────────────┘
   横断: 監査ログ（WORM・ハッシュチェーン）/ アクセス制御 / 秘密管理
```

## 2. 入力・証憑・出力

### 入力（分析対象）
- 経費精算明細
- 経費規程 / 費目 / 上限マスタ
- 従業員マスタ / 取引先マスタ
- 法人カード明細 / 購買データ
- 会計仕訳データ（拡張時）

### 証憑（実在性・妥当性の検証用、read-only）
- 領収書 / 請求書画像（OCR）
- 予定表 / メール / チャット（Microsoft Graph API）
- 経路探索 / 地図 API
- 公開情報 / 反社・制裁リスト

### 出力
| 形式 | 用途 | 主内容 |
|---|---|---|
| 統合レポート | 全体 | Exec Summary / リスク別集計 / 根拠 / 推奨アクション |
| PPTX | 経営層報告 | 注目リスク TOP3 / 費目別・部門別サマリ / リスク分布チャート |
| PDF | 内部監査・証跡保全 | 明細別 判定根拠・参照証憑リンク・違反ルール・AI所見 |
| Excel / CSV | 追加分析 | 全件リスクスコア一覧 / アラートシート / 違反ルール・寄与要因 |

すべての出力は AI の分析結果に基づき毎回自動更新。最終判断は HITL で監査人が実施。

## 3. データフロー（5フェーズ × ファネル）

```
全件データ
   │
   ▼ PHASE1 観察【選別】 ── ルール+ML を全件に決定論的適用（低コスト）
高リスク部分集合（例: 全件の 1〜5%）
   │
   ▼ PHASE2 仮説生成【認識】 ── 費目別シナリオから仮説を生成（LLM）
仮説リスト（明細ごと）
   │
   ▼ PHASE3 探索【検査】 ── Graph API 等で周辺証憑を自律収集
   ▼ PHASE4 検証【深掘】 ── 証憑と申請の整合性を照合・矛盾判定・追加探索を反復
検証済み所見（スコア・根拠・証憑つき）
   │
   ▼ PHASE5 統合【報告】 ── 全所見をレポートに統合
   ▼ L6 HITL ── 監査人が最終判断
```

### コスト設計（重要・spec改善点①）
- **LLM/エージェントは全件に流さない**。L3 のルール+ML は CPU バウンドの決定論処理で全件を捌き、推論コストの高い L4 を高リスク部分集合に限定する。
- 想定オーダー: 月次数百万明細でも、エージェント深掘り対象は数千〜数万件規模に圧縮される設計。LLM コールはこの圧縮後にのみ発生。
- スコア閾値（選別率）は `config/rules/rule_catalog.yaml` の `risk_scoring` と `thresholds` で調整可能。選別率を上げれば網羅性↑・コスト↑のトレードオフ。

## 4. データ品質・完全性（spec改善点・全件主張の裏付け）

「全件網羅」を主張する以上、**対象データの完全性**を証明できなければならない。

- **完全性照合**: 取込件数・金額合計を会計システム/ERP の管理値と突合し、欠損・重複・期ズレを検出。結果を監査ログに記録。
- **データ品質ゲート**: 必須項目欠落・型不整合・参照整合性違反を ETL 時に検出し、該当明細の所見に `data_quality` フラグを付す（除外ではなく明示）。
- **非構造化の構造化精度**: OCR の信頼度スコアを保持し、低信頼度は人手確認対象としてエスカレート。

## 5. マルチテナント・データ分離（プロダクト化時）

PwC が複数クライアントへ提供する形態では:
- テナント間のデータ完全分離（論理/物理）。クライアント横断のデータ混在を禁止。
- 「PwC のリスク検知ノウハウ」は**匿名化・一般化されたルール/シナリオ**としてのみ共有。クライアント固有データを別クライアントの学習に流用しない。
- ルールは共通ベースパック＋クライアント固有パックの二層で管理（バージョニング）。

## 6. 拡張性

本アーキテクチャは経費に限定されない。同一の「ETL→ルール/ML/探索→エージェント検証→HITLレポート」構造を、費目別シナリオとコネクタを差し替えることで以下へ展開可能:
- **会計仕訳** の異常検知（期末仕訳・休日入力・使用頻度の低い科目 等）
- **購買 / 調達** の不正（架空・分割・特定ベンダ集中・キックバック）
- **給与 / 人事** の異常（幽霊社員・残業改ざん）

拡張時は `config/rules/` にドメイン別パックを追加し、`docs/agent-design.md` のツール定義へ該当コネクタを追加する。コアのオーケストレータ・HITL・監査ログは共通。
