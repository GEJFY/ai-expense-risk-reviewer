# エージェント設計仕様 — 自律監査ループ

## 1. ループ概要

「観察 → 仮説生成 → 探索 → 検証 → 統合」を自律型 AI エージェントが反復実行する。各フェーズは状態として定義され、検証フェーズから探索フェーズへ戻る反復（追加探索）を許容する。

| Phase | 名称 | 役割 | 主処理 | コスト |
|---|---|---|---|---|
| 1 | 観察【選別】 | 全件にルール・MLを適用し高リスク経費を選別 | 決定論 | 低 |
| 2 | 仮説生成【認識】 | 費目別シナリオから仮説を生成（例:実体なき会食では?）| LLM | 中 |
| 3 | 探索【検査】 | Graph API 等で周辺証憑を自律収集（予定表/メール/移動記録/領収書）| ツール呼出 | 中 |
| 4 | 検証【深掘】 | 証憑と申請の整合性を照合し矛盾を判定、追加探索を反復 | LLM＋ツール | 高 |
| 5 | 統合【報告】 | 全所見をスコア・根拠付きでレポートに統合（HITLで最終判断）| LLM | 中 |

★ Human-in-the-Loop: AIは「評価・根拠提示」までを担い、最終的なリスク評価・対応は監査人が判断。全プロセスは自律監査ログとして記録。

## 2. エージェントのツール（すべて read-only）

エージェントが呼べるツールは以下に限定。**書き込み系ツールは存在しない**。各ツールは最小権限スコープで構成し、呼び出しを監査ログに記録する。

| ツール | 接続先 | スコープ例 | 用途 |
|---|---|---|---|
| `calendar.query` | Graph API | Calendars.Read | 当該日時に会食/出張予定があるか |
| `mail.search` | Graph API | Mail.Read | 直後の御礼/日程メールの有無、相手方一致 |
| `meeting.attendees` | Graph API | OnlineMeetings.Read | 会議出席者と申請参加者の一致 |
| `route.estimate` | 経路探索 API | — | 申請額 vs 妥当運賃の乖離（水増し疑義）|
| `geo.resolve` | 地図 API | — | 店舗所在地・業態の整合、同時刻・複数地点 |
| `ocr.extract` | 領収書OCR | — | 記載額/日付/宛名/但書の抽出と申請突合 |
| `sanctions.lookup` | 反社/制裁/公開情報 | — | 取引先の反社・制裁・休眠先該当 |
| `master.lookup` | 内部マスタ | read | 従業員・取引先・規程・上限の照会 |

### 検証パターン例（仕様）
- **交際費（会食）の実在性**: ①申請（日時・店舗・参加者・金額）から仮説生成 → ②予定表照会 → ③会議出席者と参加者一致 → ④送信メール照会（御礼/日程）→ ⑤店舗所在地・業態の整合 → ⑥矛盾あれば「実体なし疑義」をフラグ＋根拠提示。
- **旅費交通費の妥当性**: 経路探索 API × 申請額の乖離（水増し）/ 出張予定と移動記録の整合（カラ出張）/ 同一区間の定期券保有者による重複請求 / 宿泊日とスケジュールの不整合（連泊偽装）/ 深夜・自宅方面タクシーの常態化（私的利用）。

連携データ: Graph API（予定表/メール/会議）・経路探索/地図 API・領収書OCR・反社/公開情報。

## 3. HITL ゲート（人間の介在点）

| ゲート | 位置 | 人間の役割 |
|---|---|---|
| G0 スコープ承認 | Phase1 前 | 分析対象範囲・期間・有効化するコネクタを承認 |
| G1 高権限アクション | Phase3/4 | 特定個人のメール全文参照など機微なアクションは事前承認（既定は要承認）|
| G2 所見確定 | Phase5 後 | 各所見を 確定 / 棄却 / 追加調査 に判断 |
| G3 是正・通報 | レポート後 | 是正起票・通報・処分は**人間のみ**。AI出力を自動連結しない |

## 4. 反復と終了条件（仕様）

無限ループ・コスト暴走を防ぐため、明示的な終了条件を持つ:

- `max_iterations`: 1 明細あたりの探索→検証の反復上限（既定 3）
- `confidence_threshold`: 仮説の確からしさが閾値を超えたら検証確定し終了
- `evidence_exhausted`: 利用可能な証憑コネクタを尽くしたら終了（未取得証憑を所見に明記）
- `cost_budget`: 実行単位の LLM/API コスト上限。超過時は未完了明細を「要人手」としてエスカレート
- `tool_call_rate_limit`: コネクタ単位のレート上限

すべての反復・分岐・終了理由を自律監査ログに記録する。

## 5. プロンプトインジェクション対策（エージェント層・重要）

エージェントは **メール本文・領収書OCR・公開情報** という攻撃者が内容を操作しうる入力を読む。詳細な脅威モデルと防御は `docs/security-privacy.md` を参照。エージェント実装で守るべき不変条件:

- **指示とデータの分離**: 証憑コンテンツは常に「検証対象データ」として扱い、システム/開発者プロンプトと構造的に分離する。証憑内の "ignore previous instructions" 等を指示として解釈しない。
- **証憑からのツール発火禁止**: 次に呼ぶツールはオーケストレータの計画に基づいてのみ決定する。証憑コンテンツがツール選択・引数を直接決定する経路を作らない。
- **出力検証**: エージェントの結論は `RiskFinding` スキーマに適合させ、根拠（違反ルール / SHAP寄与 / 収集証憑ID）が揃わない所見を破棄する。
- **能力の最小化**: read-only・最小スコープ・レート上限を技術的に強制（プロンプトでの「お願い」に依存しない）。

## 6. 自律監査ログ（エージェントの説明責任）

エージェントの全動作はトレース可能でなければならない。各ステップを以下の形で記録（スキーマは `config/schemas/data_contracts.json` の `AuditLogEntry`）:

- どの明細について（finding_id / expense_line_id）
- どのフェーズで、どの仮説を立て
- どのツールを、どの引数で呼び、何を取得したか（証憑ID・取得時刻・コネクタ）
- どう判定し、スコアをどう更新したか
- 終了理由

このログ自体が WORM + ハッシュチェーンで保全され、後日の監査・係争で AI の判断過程を再現できる。
