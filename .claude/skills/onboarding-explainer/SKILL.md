# 非技術者への説明ガイド

目的: 聞き手が読んでいるうちに理解が深まり、技術スキルが自然に増えること。やさしくするが、正確さは削らない（＝わかりやすさは正確さへの"足し算"）。

## 説明の型

1. **たとえ → 正確な言葉** の順で話す。まず日常の比喩で直感を作り、その後に正しい用語を置く。
2. **一度に一つ**の新概念にとどめ、積み上げ式で進める。
3. **なぜ大事か**を必ず添える。「これは○○を防ぐため」と用途に結びつける。
4. 略語は初出で展開する。専門語は初出でかっこ書きの一言解説を付ける。
5. 最後に**理解の確認**を一言入れる（「ここまでで詰まる所はありますか」）。

## この領域の用語 ── たとえ付き

- **異常検知 (anomaly detection)** — 健康診断で基準値を外れた項目に印がつくのと同じ。多数派から外れた明細を機械が見つける。
- **Isolation Forest** — 「みんなと違うものほど、少ない手数で切り離せる」性質を使う手法。少数派の外れ値を効率よく見分ける。
- **HITL（Human-in-the-Loop）** — 自動運転でも最後はドライバーが判断するように、AIは候補を出すだけで、確定は必ず人間が行う。
- **プロンプトインジェクション** — 領収書やメールに紛れ込ませた「命令のふりをした文章」。それに釣られてAIが勝手に動かないようにする、という話。
- **ファネル（ふるい分け）** — 砂金採りのように、まず機械で大量を粗くふるい、残った"怪しいものだけ"を人とAIで丁寧に調べる。
- **SHAP** — 「このスコアが高いのは主にどの理由か」の内訳表。AIの判断の根拠を見える化する。
- **WORM＋ハッシュチェーン** — 一度書いたら書き換えできない記録。1文字でも変えると鎖が切れて改ざんが分かる。
- **説明可能性 (explainability)** — 「なぜそう判定したか」を必ず言えること。監査では結論だけでなく理由が要る。
- **決定論的 (deterministic)** — 同じ入力なら必ず同じ結果になること。ルールベースの部分がこれ。再現でき、監査で説明しやすい。

## 調整

聞き手の反応を見て深さを変える。専門性が見えたら用語を先に出し、比喩は最小限に。逆に不慣れそうなら比喩を厚くし、一段ずつ確かめながら進める。
